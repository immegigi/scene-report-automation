## 2. 전체 개발계획 평가

현재 계획은 관리자용 내부 MVP라는 범위 설정이 명확하다. 고객 로그인, 결제, 이메일 자동 발송, 커뮤니티 자동 가입을 제외한 점도 1차 개발 범위로 적절하다.

가장 좋은 점은 AI 기능을 한 번에 자동 실행하지 않고, 단계별 버튼으로 실행하도록 설계한 점이다. 운영자가 각 단계 결과를 확인할 수 있어 리포트 품질 관리와 위험 신호 차단에 유리하다.

다만 실제 개발에 들어가기 전 아래 항목은 보완하는 편이 좋다.

## 3. 주요 보완 필요사항

### 3-1. 관리자 인증 방식 확정 필요

1차 명령어에는 “관리자용 단순 인증 또는 Supabase Auth”로 되어 있다. 개발 시작 전 하나로 확정해야 한다.

권장안:

- MVP라도 Supabase Auth 사용
- 허용된 관리자 이메일만 접근 가능하도록 제한
- 고객 데이터와 관계 장면 내용이 민감하므로 무인증 관리자 페이지는 피할 것

### 3-2. 위험 신호 처리 기준 강화 필요

위험 신호 판별 단계는 포함되어 있으나, `caution`과 `high`의 후속 처리 차이가 더 명확해야 한다.

권장안:

- `safe`: 리포트 생성 가능
- `caution`: 생성은 가능하되 운영자 검토 필수
- `high`: 자동 생성 차단, `risk_hold` 상태 고정
- 위험 신호가 있는 입력은 PDF 생성 버튼 비활성화

### 3-3. AI JSON 저장 구조 보완 필요

현재 `analyses` 테이블은 주요 결과 필드를 담지만, AI 원문 응답과 파싱 실패 원인을 저장할 곳이 없다.

권장 추가 필드:

- `raw_ai_outputs jsonb`
- `parse_error text`
- `model_name text`
- `prompt_version text`

이 필드가 있어야 운영 중 품질 이슈를 추적할 수 있다.

### 3-4. reports 버전 관리 필요

운영자가 리포트를 수정하면 초안과 최종본의 차이를 추적할 수 있어야 한다.

권장안:

- MVP에서는 `reports`에 `updated_at` 추가
- 이후 버전에서는 `report_versions` 테이블 분리

### 3-5. PDF 생성 방식의 운영 리스크 확인 필요

Playwright/Puppeteer PDF 생성은 적절하지만, 배포 환경에서 브라우저 런타임 문제가 자주 발생한다.

권장안:

- 로컬/서버 환경 모두에서 PDF 생성 테스트
- 한글 폰트 렌더링 확인
- 5페이지 강제 분리 확인
- Supabase Storage 업로드 실패 시 상태값 변경 금지

### 3-6. 개인정보 및 민감정보 보호 설계 필요

고객 이메일, 관계 장면, 감정 서술은 민감도가 높다. 현재 스키마에는 RLS, 접근 정책, 보관 정책이 없다.

권장안:

- Supabase RLS 활성화
- 관리자 권한 정책 명시
- 다운로드 PDF 공개 URL 대신 signed URL 검토
- 삭제 또는 보관 만료 정책 추후 추가

## 4. 개발 순서 권장안

현재 소스팩 순서는 타당하나, 실제 Codex 투입 시에는 아래 순서가 더 안정적이다.

1. Supabase 스키마 정리 및 보완
2. Next.js 관리자 MVP 화면 구축
3. 관리자 인증 적용
4. 고객/제출 CRUD 및 상태값 변경 구현
5. 위험 신호 판별만 먼저 연결
6. 나머지 AI 단계별 파이프라인 연결
7. 리포트 에디터 구현
8. 품질 점검 및 승인 상태 구현
9. HTML 리포트 미리보기 구현
10. PDF 생성 및 다운로드 구현
11. 샘플 3건으로 전체 흐름 테스트

## 5. Codex 투입용 최종 명령어 압축본

아래 명령어를 새 Codex 작업 시작 시 붙여 넣으면 된다.

```text
「그 장면 해석 리포트 자동화 구축 프로젝트」를 관리자용 MVP로 구현해줘.

목표는 고객이 제출한 관계 장면을 바탕으로 관계 인지 루프와 S-C-B 패턴 코드를 분석하고, 운영자가 검토 가능한 5페이지 PDF 리포트와 이메일 본문을 생성하는 내부 관리자 도구를 만드는 것이다.

기술 스택은 Next.js, Supabase, Tailwind CSS를 사용한다. 고객용 로그인, 결제, 이메일 자동 발송, 커뮤니티 자동 가입은 제외한다.

1차로 고객 등록, 관계 장면 제출 등록, 제출 목록, 제출 상세, 상태값 관리, 리포트 에디터 틀, PDF 미리보기 틀을 만든다.

2차로 OpenAI API 기반 단계별 AI 파이프라인을 추가한다. 단계는 위험 신호 판별, 장면 구조화, 관계 인지 루프 분석, S-C-B 코드 생성, 리포트 초안 생성, 품질 점검이다. 각 단계는 한 번에 자동 실행하지 말고 제출 상세 화면에서 버튼으로 실행 가능하게 만든다.

위험 신호는 자해/자살 암시, 폭력, 협박, 스토킹, 성폭력, 미성년자 위험, 불법 요청, 상대 조종/복수 요청, 심각한 위기 표현을 감지한다. high 위험은 risk_hold로 변경하고 리포트 생성을 막는다. caution은 운영자 검토가 필요하게 한다.

S-C-B 코드는 지정된 코드 목록만 사용한다. S는 관계 장면, C는 인지 필터, B는 행동 반응이다. 결과는 pattern_code 형태로 저장한다.

3차로 운영자가 수정 가능한 리포트 에디터, 이메일 제목/본문 복사, HTML 미리보기, 5페이지 PDF 생성, PDF 다운로드 기능을 구현한다. PDF 생성 성공 시에만 submission.status를 pdf_generated로 변경한다.

데이터베이스는 customers, submissions, analyses, reports로 시작하되, AI 원문 응답과 파싱 오류 추적을 위해 raw_ai_outputs, parse_error, model_name, prompt_version 필드를 추가한다. reports에는 updated_at을 추가한다.

관리자 인증은 Supabase Auth를 사용하고, 허용된 관리자만 접근 가능하게 한다. Supabase RLS와 Storage 접근 정책도 함께 고려한다.

예쁜 UI보다 전체 흐름이 안정적으로 동작하는 MVP를 우선한다. 샘플 고객 입력 3건으로 end-to-end 테스트까지 수행해줘.
```

## 6. 최종 의견

개발 착수 가능하다. 다만 지금 상태 그대로 Codex에 넘기면 기본 MVP는 만들 수 있지만, 인증, 위험 신호 후속 처리, AI 원문 추적, PDF 생성 실패 처리, 개인정보 보호가 약하게 구현될 가능성이 있다.

따라서 개발 시작 전에는 위 최종 명령어 압축본처럼 보완 조건을 함께 넣어 지시하는 것을 권장한다.
